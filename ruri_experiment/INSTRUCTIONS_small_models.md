# 検証指示書: ruri小型モデル/静的埋め込みによる行ラベリングの精度検証

## 目的

レシートOCRパイプラインの「OCR行の埋め込みラベリング」(正規表現抽出のフォールバック候補)について、
リファレンスの **ruri-v3-310m** で実証済みの精度が、**スマホ搭載可能な小型モデルでも維持できるか**を検証する。
維持できるなら、将来 ONNX/INT8 化してパイプラインに組み込む。

最終製品の制約: スマホオンデバイス・外部依存なし・日本語・**商用利用**(ライセンス要確認)。
現在のOCR本体は INT8 で合計 11.8MB / ピークRSS 383MB。ラベラーはこれに載せる追加分なので小さいほど良い。

## 既存の資産(すべて `ruri_experiment/` 配下)

| ファイル | 内容 |
|---|---|
| `line_labeler.py` | 行ラベラー本体。数字を'0'に正規化した行テキストを埋め込み、7ラベル計41プロトタイプ文+'other'行きディストラクタ32文とのcos類似度argmaxでラベリング(閾値0.80、プレフィックスは空文字を採用) |
| `eval_labeler.py` | 一括評価。`--build-cache`でOCRキャッシュ再生成、`--sweep`で感度分析、`--prefix topic`でプレフィックス比較。評価指標は下記(A)(B)(C) |
| `ocr_cache.json` | FP32 OCR結果キャッシュ(実写9+合成20+external 40 = 69枚)。**再生成不要**(OCR側は変更されていない) |

モデル: `models/ruri-v3-310m/`(DL済み、1.2GB、Apache-2.0)。`models/` は .gitignore 済みなので新モデルもここに置く。

### 評価指標(eval_labeler.py が出力)

- **(A)** GTフィールドの正解値を含むOCR行を正しいラベルに割り当てられた率(行ラベリング単体の性能)
- **(B)** 現行 `extract_fields`(正規表現)のNGのうち、ラベラーなら回収できる件数(スイート別)
- **(C)** 現行正解を破壊する件数(誤ラベル率)

## 重要な注意

**抽出ロジック(`receipt_ocr_paddle.py`)がコミット ef9aef5 で更新されたため、過去に記録された310mの
ベースライン数値(A 291/300、B 13/44、C 3/283)は現行コードでは再現しない**(実写のregex NGが0件になった等、
(B)(C)の母数が変わる)。したがって**最初に310mを現行コードで再計測し、それを本検証のベースラインとすること**。
全モデルを同一コード・同一キャッシュで比較すれば公平な比較になる。

## タスク

1. **310mベースライン再計測**: `.venv/bin/python ruri_experiment/eval_labeler.py` を実行し、(A)(B)(C)+
   1画像あたり平均推論時間+ピークRSSを記録する。
2. **ruri-v3-30m での同評価**: `cl-nagoya/ruri-v3-30m`(HF、Apache-2.0、37Mパラメータ・256次元)を
   `models/ruri-v3-30m` にDLし、`eval_labeler.py` にモデル切り替え手段(`--model` 引数等)を追加して同評価。
   プレフィックス方式が310mと異なる可能性があるためモデルのREADMEを必ず確認し、空/推奨プレフィックスの
   両方を試して良い方を採用する。閾値0.80も30mでは最適でない可能性があるので 0.70/0.75/0.80/0.85 を掃引する。
3. **静的埋め込みモデルでの同評価**: 候補は `hotchpotch/static-embedding-japanese`(ruriを教師に蒸留された
   静的埋め込み)。**採用前にライセンスを必ず確認し、商用利用可(MIT/Apache-2.0等)でなければ不採用として
   代替(例: sentence-transformers の static retrieval 系日本語モデル)を探すか、その旨を報告する**。
   静的埋め込みはトークン埋め込みの平均なので、CPUでの推論コストがほぼゼロになる点が本命の理由。
4. **サイズ縮小の見積もり**: 最も精度の良い小型構成について、
   - PCA次元削減(310m実験では64次元でほぼ無損失だった)を32/64/128次元で試す
   - 語彙削減の余地(レシートに出る文字種は限られる)を概算する
   - 「実行時に必要なのは行埋め込み+内積のみ、プロトタイプ73文の埋め込みは事前計算可能」という前提で、
     オンデバイス搭載時の追加フットプリント(モデルサイズ・RSS・ms/行)を見積もる
5. (時間があれば)ruri-v3-30m の ONNX エクスポートを試し、変換可否と ONNX でのサイズ・速度を確認する。
   量子化までは不要。
